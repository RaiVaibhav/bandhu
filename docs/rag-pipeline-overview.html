Bandhu — Explainer

How Bandhu finds the right thing to say

Bandhu never lets the AI invent advice from scratch. It always searches a small, human-reviewed library first, and only uses what it finds there. This is called a RAG pipeline — short for Retrieval-Augmented Generation: "look something up, then write about it," instead of "just make something up."

Everything below happens in two very different phases: one that happens once (and gets updated occasionally), and one that happens every single time someone checks in.

The building blocks

Ten pieces, in plain language. Each one shows up in the diagram below.

Content library

The trusted material Bandhu is allowed to draw from — short, warm entries adapted from WHO and NIMHANS counseling guides, written and checked by people, never invented on the spot by the AI.

Chunking

Breaking that material into small, bite-sized entries — a sentence or two each — so the app can pull exactly the right piece instead of a whole page.

Tagging

Labeling each entry with what it's about: which feeling, which topic, how intense, which language — so the search can be narrowed before it even starts.

Embedding

Turning a piece of text into a list of numbers that captures its meaning. This is what lets the app find "close enough" matches, not just exact keyword matches.

Vector database

The storage system holding every entry's numbers, ready to be searched instantly. Bandhu uses one called pgvector.

Retrieval

The moment the app actually searches that storage for the 2-3 entries that best match what the person just said.

Memory (light)

A short running summary of someone's recent check-ins — moods, what's helped before — never a raw transcript of everything they've said.

Orchestrator

The small decision-making moment that decides whether to gently offer something this time, or just acknowledge and stop — quietly, not as a default.

Guardrails

Automatic checks run on every reply before a person ever sees it: is it short enough, is it safe, does it avoid sounding like medical advice.

Evaluator

A periodic quality check — a sample of replies get scored against a real counseling rubric (MITI) to make sure the tone stays warm over time, not just on day one.

How it fits together

Top row happens rarely. Bottom row happens on every message. The knowledge base built in the top row is what gets searched in the bottom row.

A Building the knowledge base — done once, refreshed occasionally
📚Vetted sources
✍️Human rewrite & review
🏷️Tag by feeling & topic
🔢Turn into numbers (embed)
🗄️Store in the vector database
this is the library the search below reaches into
B Answering a check-in — happens every single message
💬Person's message
🛟Safety check
🔎Understand what it's about
🧵Recall recent memory
📖Search the knowledge base
🤔Decide what to say
✏️Write the reply
✅Check the reply
📤Send it & remember

Underneath all of this: every step in row B is recorded (via Phoenix tracing) so the team can see exactly what happened for any single check-in — which entries got retrieved, what got said, whether the checks passed — without guessing after the fact.

Why split it this way? Building the knowledge base is slow, deliberate, and reviewed by people — it should be. Answering a check-in has to be fast and cheap, because it happens on every message, for every person, indefinitely. Splitting the work this way means the expensive, careful part only happens once, and the part that repeats forever stays light.
