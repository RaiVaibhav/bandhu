What changed, and why

Two inputs forced this revision

1. "Stop suggesting things — this is a companion"

v0.1's orchestrator treated offer_suggestion, name_thinking_trap, and close_the_loop as three peers, capped at one per turn. That still made "offer something" the orchestrator's default mode of operation. The correction: the acknowledgment has to be able to stand alone as the whole product, and offering has to be the exception, not a coin-flip every turn.

2. The live Stitch prototype confirmed it, visually

The Response screen forces a two-button decision ("Not right now" / "Yes, let's try") right after every acknowledgment, and the Thinking Trap screen shows a checklist of all 8 distortions instead of one guess. Both are the UI expression of the same backend problem: the pipeline was built to offer by default and stay quiet by exception, which is backwards.

v0.2's actual change: a new deterministic eligibility gate before the orchestrator ever gets to make a suggestion, plus a split between two kinds of optional behavior that were wrongly treated as the same thing in v0.1.

Care vs. help — not the same optional

Re-reading the README while fixing this surfaced a distinction v0.1 flattened. close_the_loop — referencing something from a past check-in — is explicitly what the spec calls "what makes the memory feel like care rather than tracking." That's not a suggestion, it's the companion behavior itself. offer_suggestion and name_thinking_trap are genuine help-offers — useful, but secondary, and the thing that needed rationing. v0.2 treats them differently.

Flow diagram

11 stages · 4 branches
1
Deterministic

Ingest & normalize

Text, voice-to-text, image caption, or a mood-tap value. Detects language (incl. code-mixed). Image inputs get a document-type check first.

2
Deterministic

Safety gate

Rules-based crisis-language scan, reads current message + recent context so a hedge can't suppress a true positive.

3
Deterministic

Classify

Special-case flag (medical / legal / disorder / dependency / medication / minor-mode) + emotion·category·intensity tag.

4
Deterministic

Memory read

Light structured summary — recurring themes, last suggestion + outcome — plus a rolling help-offer count (new in v0.2): how many of the last N check-ins included a suggestion or thinking-trap.

5
Eligibility gate — new

Is a help-offer even allowed this turn?

Deterministic, not a model decision. Default: no. Becomes eligible only if the tag shows genuine negative/high-intensity signal and the rolling help-offer count is under the cap (default proposal: no more than 1 in the last 3 check-ins). Gates offer_suggestion and name_thinking_trap only — never close_the_loop.

6
Deterministic

Retrieval

Runs whenever there's a tag, cheap either way — prepared in case the orchestrator ends up eligible and chooses to use it. Metadata-filtered, top 2-3 chunks.

7
LLM judgment

Orchestrator

Two-tier, not three peers. Silence is a valid, expected outcome even when eligible.

close_the_loop? — always considered offer_suggestion? — only if eligible name_thinking_trap? — only if eligible, exactly one pattern, phrased as a guess
8
Deterministic

Generate

Acknowledgment first, written to be complete on its own. At most one quiet line appended — close_the_loop takes priority over a fresh help-offer if both would apply. Rendered as an easy-to-ignore line, never a two-button decision card.

9
Deterministic

Guardrail check

Length, banned diagnostic/medical/legal phrasing, no-advice-giving — plus (new): reject if more than one distortion label appears, reject if the optional line reads as a forced choice rather than a dismissible aside.

10
Deterministic

Memory write → response

Update mood/theme. Increment the rolling help-offer count only when a help-offer was shown — close_the_loop doesn't count against the cap.

11
Async · sampled

Sampled evaluator

MITI-based tone score, 5-10% of responses — plus (new) a "does the acknowledgment read as complete on its own" axis, scored independent of whether anything was offered.

Crisis response

Fixed template, bypasses everything else. Shown once per conversation.

Special-case redirect

Fixed template per category, rejoins at Guardrail check. Unchanged from v0.1 — still needs the six templates written (see status below).

Not eligible → skip straight to Orchestrator

Retrieval still runs (cheap), but the orchestrator only sees close_the_loop as an option. No fallback messaging needed — plain acknowledgment is a complete, correct outcome, not a degraded one.

Fallback safe response

Guardrail fail → acknowledgment only, no optional line, logged for review.

v0.1 findings — status

Resolved by v0.2

Orchestrator one-tool-per-turn rule

Superseded by something stricter: a deterministic eligibility gate plus an explicit close_the_loop-over-fresh-offer priority rule, rather than a soft cap left to model judgment.

Still blocks build

Six special-case redirect templates

Medical doubt, life decisions, disorder questions, dependency signals, medication questions, medical-document images. Unaffected by this revision — still a content-library gap.

Still blocks build

Safety gate needs conversation memory

Latest-message-only scoring still misses the hedge-after-crisis case. Independent of the suggestion-frequency work.

Still blocks build

Image ingest must reject medical documents pre-caption

Unchanged — still needs to be an ingest-time check, not a policy statement.

Decision needed

Help-offer frequency cap — default proposed, not validated

New in v0.2: "no more than 1 help-offer in 3 check-ins" is a starting number, not a researched one. Needs real usage data or at least a product call before it ships as-is.

Decision needed

Minor / age-unknown session flag

Unchanged from v0.1 — still needs the underlying policy decision; plumbing is ready either way.

Decision needed

Low-confidence tags need an honest fallback

Reinforced by the default-to-silence stance but not fixed by it — Classify still needs to be able to say "not enough signal" explicitly.

v0.2 — the pipeline shape is now backend-correct for "companion first." It doesn't fix the Stitch screens on its own: Response's two-button fork and the Thinking Trap checklist are UI decisions that need their own pass (see the design audit) even though the backend behind them has changed.
